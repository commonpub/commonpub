The CommonPub Protocol

An open ActivityPub federation protocol and package suite for self-hosted maker communities. Projects and hacks are relics of making — displayed, catalogued, and shared across independent instances without a central authority.

Cross-Site Auth — How Login Works Across Instances NEW v4.5

The baseline answer: each CommonPub instance is its own auth domain. A hack.build account is not a deveco.io account. Separate email, separate password, separate session. This is the correct default — instances are sovereign. The question is what you optionally layer on top.

Three Models — Pick Per Instance

MODEL A — INDEPENDENT (DEFAULT)
One account per instance. No shared state.
Each instance runs its own Better Auth. You register separately on hack.build and deveco.io. Federation between instances works through AP activities — you can follow, like, and fork across instances without logging in to the remote site. This is exactly how Mastodon works: you don't log in to mastodon.social to follow someone there. Your home account does the interacting.

Most third-party deployers should use this. Zero operational complexity. No shared infrastructure. No auth dependency chain.
default zero shared infra sovereign instances
MODEL B — AP ACTOR SSO (V1 FEATURE)
"Sign in with your CommonPub actor" — authenticate on a remote instance using your home account
This is for the case where you want a real session on a remote instance — not just federated interaction, but actual write access (creating posts, managing a community, editing a profile). The flow is OAuth2-based, using the AP actor identity as the identity provider:
User visits deveco.io login page ↓ Clicks "Sign in with CommonPub actor" ↓ Enters actor handle: @moheeb@hack.build ↓ @commonpub/auth resolves the handle via WebFinger → gets actor JSON → gets oauth_endpoint ↓ Redirects to hack.build/oauth/authorize?client_id=deveco.io&scope=identity&redirect_uri=... ↓ User authenticates on hack.build (their home instance — normal session) ↓ hack.build returns signed JWT: { sub: "@moheeb@hack.build", ap_id: "https://hack.build/users/moheeb", ... } ↓ deveco.io receives token, verifies signature against hack.build public key (from actor JSON) ↓ deveco.io creates a local "federated user" record — no password stored, ap_id is the foreign key ↓ Session established on deveco.io with federated_user role (limited permissions by default)
The remote instance stores a federated_users shadow record: no password, no email (unless shared), just the verified AP actor ID and a local session. Permissions for federated users are scoped — they can join communities, comment, and post, but can't access admin features unless an admin explicitly grants a role.

This is how Moheeb logs into deveco.io with his hack.build account. One real account at home, shadow sessions anywhere else.
AP actor as identity OAuth2 flow no shared DB built into @commonpub/auth
MODEL C — SHARED AUTH DB (OPERATOR OPTION)
For operators who own multiple instances and want a single unified account
An operator running both hack.build and deveco.io can point both instances at the same DATABASE_URL for the Better Auth tables only — users, sessions, accounts — while keeping separate content databases. This requires a separate shared Postgres cluster for auth and explicit opt-in in commonpub.config.ts: 
// commonpub.config.ts — shared auth across operator-owned instances

defineCommonPubConfig({
  auth: {
    sharedAuthDb: process.env.SHARED_AUTH_DB_URL,  // separate DB from content
    trustedInstances: ['hack.build', 'deveco.io'],  // list of co-owned instances
  },
})
When sharedAuthDb is set, @commonpub/auth initializes Better Auth against that URL instead of DATABASE_URL. The content DB stays separate — content_items, communities, federation tables all live in the instance-specific Postgres. Only the users/sessions/accounts tables are shared. Migrations for auth tables run against the shared DB; content migrations run against the local DB.

This is purely an operator convenience pattern. It breaks the "fully independent instance" model intentionally, and that's fine — the operator knows they're coupling their instances. Third-party deployers should never use this. The commonpub.config.ts validation logs a warning when this option is set.
operator opt-in only breaks instance sovereignty auth DB shared, content DB separate

The mental model: each CommonPub instance is its own country. Model A = you live there. Model B = you're a visitor with a valid passport from your home country. Model C = dual citizenship (operator special case). Federation (follows, likes, forks) works without any of these — it's postal mail between countries.

Communities NEW v4.5

Communities are the social layer of a CommonPub instance. They are topic-organized spaces where members collect, discuss, and share content. hack.build communities are technology/interest groups (ESP32 Builders, Open Source Hardware, Phoenix Makers). deveco.io communities are platform-organized (ESP32 community, Raspberry Pi community, RISC-V). Each community has a feed, membership with roles, and optional federation as an AP Group in v2.

Community vs Hub — Distinction Closed

"Hub" is retired. Prior versions of the plan used "Hub" — this created ambiguity (is a hub a geographic area? a topic? a team?). The concept is unified under Community. A community is a persistent group with a feed, members, and admins. It lives at /c/[slug]. The Better Auth "organizations" table is NOT used for communities — org is reserved for multi-user authoring teams. Communities are their own schema.

Content Feature Integration

Communities integrate with the rest of the content system in a few specific ways:

Sharing into a Community
Any content_item (project, guide, explainer, article, blog post) can be shared into one or more communities by its author. The share creates a community_posts row with post_type = 'share' and content_item_id set. The original content_item is not duplicated — the community post is a reference card with an optional author comment. A content_item tracks which communities it's been shared to via a community_shares join table.
reference, not copy multi-community
Community-Scoped Search
Meilisearch and Postgres FTS both support filtering by community_id. When a user searches from inside a community, the default scope is that community. A toggle expands to site-wide. community_posts are indexed separately from content_items — different fields, different ranking rules. The DocsSearch component can similarly be scoped to a community's associated docs site.
community-scoped default expandable to global

Moderation Tools

Post Actions
Pin / unpin (admin+). Lock thread (mod+). Remove post with reason (mod+). Removed posts show "[removed]" placeholder rather than hard delete — preserves thread structure. Author can delete their own posts if no replies exist.
Member Actions
Approve/deny join requests (admin+). Remove member (admin+). Temporary or permanent ban with reason (mod+). Promote/demote roles up to but not including owner (admin). Transfer ownership (owner only).
Spam / Abuse
Report post (any member). Reports queue in /c/[slug]/settings/reports. No global moderation queue in v1 — each community moderates its own space. Instance admin can see all reports across all communities in /admin/reports.

Federation Path — AP Group (v2)

Communities do NOT federate in v1. This is deliberate. AP Group support requires non-trivial work: the community becomes an AP actor, posts become Group activities, Announce semantics differ from individual actor Announce, moderation becomes cross-instance. Getting community federation wrong (like Lemmy's early bugs) poisons trust. v1 communities are local. v2 federation only after v1 communities have real usage and real moderation experience.
v2 AP Group flow (future): @esp32-builders@hack.build ← AP Group actor with inbox + outbox + followers collection Member posts to community → Create(Note) sent to Group inbox Group verifies membership → Announce(Create(Note)) to all Group followers Remote instance follows Group → receives all Announces → renders in local community feed Remote user joins local Group → AP Follow to Group actor → approved → gets Announces deveco.io user joins esp32-builders@hack.build: @pcb_hacker@deveco.io sends AP Follow to @esp32-builders@hack.build hack.build community admin approves (or auto-approves if join_mode = open) deveco.io receives Announces from the Group — renders in deveco.io's federated feed

Theming — CSS Custom Properties Only

Every @commonpub/ui and @commonpub/docs component references only var(--*) tokens. A complete visual rebrand is a single tokens.css file. No CSS-in-JS, no build step, no runtime overhead. Community components follow the same contract — CommunityCard, PostCard, PostComposer all inherit site tokens.
@commonpub/ui/theme/base.css ← full token surface, sensible defaults @commonpub/docs/theme/docs.css ← docs-specific tokens, inherits base ↓ overridden by ↓ [your-app]/src/theme/tokens.css site-level overrides docs_sites.theme_tokens (DB column) per-docs-site overrides hack.build: Permanent Marker display, Special Elite body, VT323 mono, paper/ink palette deveco.io: system-ui, pink/teal/yellow on white third-party: whatever tokens.css says — zero component code changes needed
