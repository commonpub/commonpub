## THE decision: FormKit abandoned for the auto-form

The kickoff baked in **FormKit + `@formkit/zod`** per
[[feedback-phase-3-hybrid-libraries]] and said "NO path-pick". But the
kickoff ALSO carried a hard rule — *"verify load-bearing claims against
source"* — and that rule overturned the library choice before a single
line was written on top of it:

1. **`@formkit/zod` is validation-only.** Confirmed against FormKit's own
   docs (`formkit.com/plugins/zod` + `/api-reference/formkit-zod`):
   `createZodPlugin(schema, cb)` returns `[plugin, submitHandler]` and adds
   Zod VALIDATION to a form whose inputs you **author by hand**. It does
   **not** generate fields from a schema. It delivers none of Phase 3e's
   auto-gen goal — you'd hand-write all 17 section forms anyway.
2. **Zod-major mismatch.** `@formkit/zod@2.0.0` (latest, 2026-04-02)
   peer-deps `zod@^3.0.0`; the monorepo is uniformly on `zod@4.3.6`
   (schema package + layer + server validators). FormKit hasn't shipped
   Zod 4 support.
3. **Theme cost.** Genesis theme is incompatible → a full custom
   `@formkit/themes` layer = a brand-new design-system CSS surface, exactly
   the cascade-bug magnet [[feedback-css-cascade-unit-test-blind-spot]]
   (session 166's P0 lesson) warns against.

**Surfaced to the user via AskUserQuestion** (it contradicted a deliberate
baked-in directive on a 2-3 session arc). User chose the recommended
**hand-rolled `<AutoForm>` driven by Zod 4's native `z.toJSONSchema()`** —
which is the plan's own risk-register fallback ("fall back to a hand-rolled
`<AutoForm>` — Phase 3e decision"). Recorded in
[[project-session-167-formkit-pivot]].

Empirically verified `z.toJSONSchema()` against all 17 real schemas: emits
clean, complete field metadata (type/enum/anyOf-const/minLength/maxLength/
pattern/minimum/maximum/minItems/maxItems/default/nested items+properties/
required). Maps 1:1 onto the §7.10 input table. **Zero new bundle weight.**

## What shipped (4 commits)

### `a5cd53d` — 3e.1: pure Zod→form engine

`composables/autoFormSchema.ts` — framework-free (no Vue). `buildAutoForm(schema)`
→ `{ fields: AutoFormField[], isEmpty }`. Walks `z.toJSONSchema()`:

| JSON Schema node | Control |
|---|---|
| `string` (no pattern, maxLength < 480) | text |
| `string` (maxLength ≥ 480) | textarea |
| `string` + `pattern` (URLs/paths) | text — NOT `type=url` (rejects valid root-relative `/about`); pattern carried for validation |
| `enum` | select |
| `anyOf:[{const}…]` (union of int literals — heading.level, columns) | select (numeric coercion on the way out) |
| `integer`/`number` | number stepper (`min`/`max`/`step`) |
| `boolean` | toggle |
| `array<object>` | repeater (item sub-fields + blank-item default) |
| `object` | group (recursive) |
| empty object (stats) | `isEmpty: true` |

Form-`required` = in `required` AND no default (a defaulted field is
pre-filled, never asterisked). `optional` = not-required AND no default
(drives the "— Default —" select option, see R1-1). Defensive `$ref`/`$defs`
resolution + try/catch degrade-to-empty on unrepresentable schemas. Kept
PURE specifically so the only place a CSS-cascade bug can hide is the `.vue`
(per the css-cascade-blind-spot lesson). 15 table tests over all 17 builtin
schemas + the row schema.

## Self-audit (R1-R4 + fresh-eyes + audit-of-audit)

The architecture choice paid off here: keeping the engine pure + reusing the
proven native-input CSS (vs a new FormKit theme) **structurally avoided** the
CSS-cascade-bug class that produced session 166's round-2 P0. Lower yield is
expected + correct, not a sign of a shallow sweep.

| # | Sev | Finding | Disposition |
|---|-----|---------|-------------|
| R1-1 | P2 | Optional enum selects with no default (row gap/align/paddingY) rendered the FIRST real option as visually selected while the value was `undefined` — misleading (renderer applies its OWN 'md' default). | **FIXED** `1cffcd1` — engine `optional` flag; optional selects render a leading "— Default —" option; undefined binds to it; choosing it clears the key. |
| R4-1 | P2 | `fieldset` defaults to `min-inline-size: min-content` → horizontal overflow in the 320px inspector. | **FIXED** `1cffcd1` — `min-width: 0` on the array/group fieldsets. |
| R2-1 | — | `<select :value>` reflects the current value on render? | **Verified clean** — added a regression assertion (heading.level=3 → select shows "3"). |
| R2-2 | — | Invalid config → save error toast? | **No bug by design** — `validators.ts:798` validates section config as `z.record(z.string(), z.unknown())` (per-type enforcement still DEFERRED since session 160). Server accepts any config → no error toast; inline validation is correctly ADVISORY. |
| R3-1 | — | Client/server schema drift | Clean — both import per-type schemas from `@commonpub/schema` (one source). |
| FE-CSS | — | Does any `pointer-events`/cascade rule hide the row-select handle (the session-166 P0 shape)? | **Traced clean** — only the resize OVERLAY sets pointer-events:none; no `cpub-layout-row--editable > *` rule exists; the section-level rule applies only to section children. Row-select is a direct row child like the working remove button. |
| R3-2 | P3 | Schema-version drift shows an advisory but doesn't APPLY `def.migrations`. | Defer — migration application is Phase 3f; advisory is the cheap correct signal. |
| R1-2 | P3 | Clearing a number input snaps back to its default (valueOf falls back). | Defer — safe + rare; all builtin numbers have defaults. |
| R1-3 | P3 | markdown/custom-html long text uses a plain 4-row textarea (no rich editor). | Defer — §7.10 wants TipTap via `.describe('rich')`; no schema uses describe yet (next-session rich-field seam). |

**Recursion-curve**: 2 real fixes + 1 verified-clean + 6 reasoned-clean/defer.
