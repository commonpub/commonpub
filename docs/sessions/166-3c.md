### Library decision — vanilla over grid-layout-plus

The session-163 collision-boundary deferral held up: a parallel
Explore-agent audit verified `grid-layout-plus@1.1.1`'s `grid-item.vue:720`
attaches interactjs to the SAME wrapper element `@vue-dnd-kit/core`
already owns for the drag-section gesture. Compounding factors:

- `grid-layout-plus` uses **absolute positioning + transforms** to size
  items; our sections live in a CSS `display:grid` row via
  `grid-column: span var(--cpub-section-cols-lg)`. Incompatible sizing
  models on the same children.
- Coordinate model (x/y/w/h) doesn't match our `colSpan` — needs
  propMap shim across 17 built-in sections.
- ~40-60h with rework + collision mitigation vs ~15-25h vanilla.

PATH Y owns a SEPARATE child element (the right-edge handle), sidestepping
the collision entirely. Adopted; closes the §163 deferred risk.

### Round-2 audit (ultrathink fresh-eyes pass) — 3 findings

User requested an `ultrathink audit` after the round-1 close. The
slower careful re-read with distance from the code caught what the
rapid R1-R4 sweep missed. **The "two consecutive 0-finding rounds"
claim from round 1 was wrong** — the sweep moved too fast to catch a
CSS-cascade specificity issue.

| # | Severity | Issue | Fix |
|---|----------|-------|-----|
| RD2-1 | **P0** (shipping-blocking) | `.cpub-layout-section--editable > *:not(.cpub-layout-section-moves) { pointer-events: none }` cascade made the resize handle **HIT-TEST-INVISIBLE in a real browser**. Tests passed because jsdom's `dispatchEvent` bypasses pointer-events hit testing. Feature did not work in production. | Extend the `:not()` chain to also exclude `.cpub-layout-section-resize-handle` |
| RD2-2 | P1 | `resizable: false` on a neighbour was treated as resizable — absorbing the source section's delta into a fixed-width neighbour silently violated the operator's opt-out. Affects hero + divider builtins. | Both pointer (LayoutRow) + keyboard ([id].vue lookupResizeBounds) paths: treat `neighbourDef.resizable === false` as `neighbourMin = neighbour.colSpan` (effective fixed wall) |
| RD2-3 | P2 | `section.responsive.lg !== null` makes the LG viewport render `responsive.lg` instead of base; mutating base in that case is invisible at LG. v1 ships with no UI to author `responsive` + no built-in section sets it (latent). | Documented in `useLayoutResize.ts` header so Phase 3e knows where to plug in per-breakpoint editing |

**Lesson learned**: rapid R1-R4 sweeps catch local issues; CSS cascade +
specificity bugs need a different lens (literal stylesheet trace +
real-browser verification). The PointerEvent jsdom shim
([[feedback-jsdom-pointerevent-missing]]) compounded this — the same
shim that enabled my unit tests is also why those tests can't catch
production-CSS-cascade interactions.

**Future hedge**: write at least one resize test that mounts in a real
browser via `@nuxt/test-utils` + Playwright. Currently scheduled for
the Phase 3e arc where the editor-shell-wide axe scan needs the same
harness. Documented in `167-kickoff.md` as a Path A sub-task.

## Decision log

1. **PATH Y (vanilla pointer events) over PATH X (grid-layout-plus)** —
   explore-agent verified the session-163 collision risk; CSS Grid model
   preserved; 60% cheaper.
2. **Module-scoped singleton state** matching useLayoutHistory + announcer
   pattern. NOT a Pinia store — same rationale documented at
   `useLayoutHistory.ts:9-17` (no Pinia surfaces in the monorepo).
3. **Closure-injected lookupResizeBounds + getDraft** rather than direct
   imports inside the composable. Mirrors existing `getDraft` /
   `getSelection` shape in `useLayoutHotkeys` and keeps the composable
   framework-agnostic.
4. **Section's right neighbour absorbs** the inverse delta (vs the
   row redistributes). Plan §7.5 explicit; the sum-invariant + "stop
   cold at neighbour-min" rule keeps the row total constant.
5. **No mobile inspector slider this session** — plan §7.5 mentions it
   but Phase 3e ships the inspector form. For Phase 3c, mobile users
   have the Shift+Arrow keyboard path; touch users see a hidden handle
   but a non-functional one would be more confusing than absent.
