My Journey Building a Voice Assistant with Raspberry Pi

TK
Tomasz Krawczyk
@tkrawczyk_builds · Feb 28, 2026 · 9 min read · DIY Smart Home · Part 2 of 4
5,841 views

So. I did it. After three weekends of fighting with audio drivers, one accidental rm -rf incident that I'd rather not discuss, and approximately eleven cups of coffee at 11 pm, I have a working voice assistant running entirely locally on a Raspberry Pi 5. No cloud. No subscriptions. No Google listening in on my kitchen conversations about whether we're out of oat milk.

This is part two of my DIY Smart Home series. In part one, I covered the home automation backbone — MQTT broker, Node-RED flows, and the basic sensor setup. This time it's all about voice. Specifically, getting Whisper for speech recognition, Piper for text-to-speech, and a Rhasspy-style intent pipeline stitched together into something I'm not embarrassed to show people.

Why Not Just Use a Smart Speaker

This is the question my partner asked, very reasonably, when I showed up with a USB microphone array and a look of grim determination on a Saturday morning. And she's right — you could absolutely use a Google Home or an Echo. It would work better, immediately, with no effort. I get it.

But I've got a few reasons. First, privacy. I really don't want recordings of every conversation in my home going to a data centre. Second, reliability — my assistant doesn't need internet access to turn off the living room lights. Third — and I'll be honest about this — I just wanted to build it. That's reason enough.

There's a specific kind of satisfaction that comes from having hardware in your home that you understand completely. Every part of it, from the kernel audio driver to the intent handler, is something I put there. That's not nothing.

The Hardware Setup

I'm running this on a Raspberry Pi 5 with 8 GB RAM. The Pi 5 is genuinely fast enough for Whisper's small model at acceptable latency — I'm getting around 800ms end-to-end from end of speech to intent trigger, which feels responsive enough for home automation commands.

For audio input I went with a ReSpeaker 4-Mic Array USB hat. It does hardware beamforming and has good noise rejection, which matters a lot in a kitchen environment where there's fan noise and occasionally a very loud espresso machine. I tried a basic USB webcam mic first and it was genuinely terrible — the keyword false-positive rate was embarrassing.

For audio output I'm using a pHAT DAC with a small passive speaker. The TTS output doesn't need to be hi-fi — it just needs to be intelligible across the room. Piper's voices are surprisingly good for a local TTS engine; I'm using en_US-lessac-medium which sounds natural enough that my kids have started saying please and thank you to it, which is either heartwarming or dystopian.

The Software Stack

Here's the full pipeline, roughly in order of execution:

  1. Openwakeword — lightweight keyword detection, running continuously at very low CPU cost. Watching for "Hey Computer" (yes, I know).
  2. Silero VAD — voice activity detection to capture the audio segment after wake word confirmation.
  3. Whisper small.en — offline speech-to-text. Runs on CPU via faster-whisper, which is significantly more efficient than the vanilla Python implementation.
  4. Custom intent parser — a simple pattern matcher I wrote in Python. More on this below.
  5. Node-RED integration — sends MQTT commands to the home automation layer.
  6. Piper TTS — generates the spoken response.

The whole thing is orchestrated by a Python service that runs on startup via systemd. Let me show you the core audio capture and transcription loop, because this took me embarrassingly long to get right:

Python voice_pipeline.py
import asyncio import numpy as np from faster_whisper import WhisperModel from openwakeword.model import Model as WakewordModel # Load models once at startup — takes ~4s, acceptable for a service whisper = WhisperModel("small.en", device="cpu", compute_type="int8") wakeword = WakewordModel(inference_framework="onnx") async def capture_and_transcribe(audio_stream): """ Wait for wake word, capture utterance, return transcript. Blocks until a complete utterance is detected. """ async for chunk in audio_stream: scores = wakeword.predict(chunk) if scores["hey_computer"] > 0.75: play_ding() # audible confirmation utterance = await capture_utterance( audio_stream, max_silence_ms=1200, max_duration_s=8 ) segments, _ = whisper.transcribe(utterance, beam_size=1) transcript = " ".join(s.text for s in segments).strip() if transcript: yield transcript

A few notes on the implementation choices: compute_type="int8" is crucial — it cuts Whisper's memory usage roughly in half and speeds up transcription by around 40% with negligible accuracy loss on clean speech. beam_size=1 is greedy decoding; you lose a little accuracy versus beam search but it's meaningfully faster for short commands where the model is rarely uncertain anyway.

The Intent Parser: Not ML, Just Good Old Pattern Matching

I spent a day trying to use a local LLM for intent parsing and gave up. It was slow, unpredictable, and I kept getting responses like "Certainly! I'd be happy to help you turn off the living room light!" when I just needed a JSON object. Instead I wrote a dead-simple regex-based parser, and honestly? It works great for the ~40 commands I've defined.

Python intent_parser.py
import re from dataclasses import dataclass from typing import Optional INTENTS = [ (r"turn (on|off) (?:the )?(.+?)(?: light)?$", "light_control"), (r"set (?:the )?(.+?) to (\d+)(?: percent)?$", "set_level"), (r"what(?:'s| is) (?:the )?(?:temp|temperature)", "query_temp"), (r"(lock|unlock) (?:the )?(?:front )?door", "door_control"), ] @dataclass class Intent: name: str entities: dict def parse_intent(text: str) -> Optional[Intent]: text = text.lower().strip(" .") for pattern, intent_name in INTENTS: match = re.search(pattern, text) if match: return Intent( name=intent_name, entities={"groups": list(match.groups())} ) return None # Unknown intent — say "sorry, try again"
Tip: Test Whisper transcription variance first
Before writing your intent patterns, collect 20–30 real transcriptions of your target phrases. Whisper is consistent but not identical run-to-run, especially with room noise. Build your regexes against the actual output you observe, not how you think you'll phrase things.

What Actually Went Wrong

Let me document the failures because tutorials that skip this are lying to you.

ALSA vs PulseAudio hell. I spent four hours on this. The ReSpeaker shows up as a USB audio device and PulseAudio's default configuration fought with the direct ALSA access that OpenWakeWord wants. The fix was to run the wake word detection process with PULSE_SERVER="" to bypass PulseAudio entirely for that one process. Not elegant, works fine.

Whisper CPU heat. Under sustained use, the Pi 5 CPU gets hot and throttles. I added a heatsink and a small 40mm fan to the case and the problem went away. Don't skip the heatsink on a Pi 5 if you're running ML inference.

False positives at weird hours. The television. Specifically, one episode of a documentary where someone said something that phonetically resembled "hey computer" and I woke up at 2 am to my assistant saying "sorry, I didn't understand that." I lowered the wake word threshold slightly and added a time-of-day gate: between midnight and 7am it requires a higher confidence score to trigger.

The moment your creation wakes you up at 2am to say it doesn't understand the television, you have crossed a threshold from maker project to haunted house.

Watch out: microSD write endurance
If you're logging audio snippets for debugging (which you should be, at least while tuning), make sure you're writing to an external SSD or a RAM disk rather than the microSD. Whisper debug logging can be surprisingly write-heavy and microSD cards don't love sustained writes.

Where It Stands Now

After about three weeks of daily use, it works reliably enough that I've stopped thinking about it as a project and started using it as an appliance. Command recognition rate is around 94% in normal conditions, dropping to maybe 85% when there's significant background noise. For a fully local, free system running on a $90 computer, I'm satisfied with that.
